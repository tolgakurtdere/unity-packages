# com.tk.core 0.5.0 — Startup settings (display + logs) — Design

Approved by Tolgahan 2026-07-22 (brainstorm; scope locked to a "startup policy" asset covering display **and** log suppression, with the added requirement that test builds keep their logs). Driver: game-shikaku shipped its first Android build and felt sluggish. On-device measurement (Redmi Note 12 Pro 5G, 120 Hz panel, Android 14) showed a flat 30.0–30.5 FPS across 155 s of play — Unity leaves `Application.targetFrameRate` at the mobile default of 30, and `QualitySettings.vSyncCount` is ignored on Android, so no quality level ever raised the ceiling. A local `DisplayBootstrap` (targetFrameRate 60 + `NeverSleep`, applied at `BeforeSplashScreen`) moved it to 60.5 FPS / 16.53 ms mean frame time. Every game built on `com.tk.core` needs this, so it belongs in the package rather than being re-typed per project.

## Why not the obvious placement

- **Timing.** `AppBootstrapper` is a splash-scene `MonoBehaviour`; its `Start()` runs after the engine splash is already up, so the splash and the first frames keep the platform default. The setting must be applied at `RuntimeInitializeOnLoadMethod(RuntimeInitializeLoadType.BeforeSplashScreen)`.
- **No scene exists at that moment.** Core's existing SO convention (`UICatalog`, `TabBarConfig` assigned in the inspector) *structurally cannot* deliver a value at `BeforeSplashScreen` — there is no scene and no object to hold the reference. The config has to be reachable without a scene.
- **Scope.** A frame rate is project-wide platform policy, not a property of one scene object. In scene YAML it is invisible in review, merge-conflict prone, and only applies when that scene boots first.
- **Latent bug found while designing this.** `AppBootstrapper.disableLogsInReleaseBuilds` is applied under `#if !UNITY_EDITOR`, which also covers **development** builds — despite its name it kills logs in test builds too. That is why `com.tk.haptics`' Android `SecurityException` warning was invisible even in a development build.

## Locked decisions

1. **Delivery: a Resources-loaded ScriptableObject.** `Resources.Load<StartupSettings>("TKStartupSettings")` inside the `BeforeSplashScreen` hook. Chosen over *Preloaded Assets* (consumer can forget to register it → silently no-ops, the exact failure mode we are fighting; load order vs. `BeforeSplashScreen` is not guaranteed) and over *editor codegen* (too much machinery for four fields). Cost accepted: one `Resources/` folder in the consumer project.
2. **Scope: one "startup policy" asset**, covering display *and* log suppression — not display-only. The log flag's late-timing bug cannot be fixed while the flag lives on a scene `MonoBehaviour`, so fixing it *requires* moving it. One consistent home for startup policy.
3. **Frame rate: four modes × two platform profiles.** `PlatformDefault | Fixed | MatchRefreshRate | HalfRefreshRate`, held per profile (`mobile`, `standalone`), selected at runtime by `Application.isMobilePlatform`. `MatchRefreshRate` exists because 60 does **not** divide evenly into a 90 Hz panel and beats against the refresh; `HalfRefreshRate` is the battery mode. Two profiles (not per-OS) — mobile-vs-desktop is the real split; finer granularity bloats the API with no consumer.
4. **Sleep timeout: tri-state.** `LeaveDefault | NeverSleep | SystemSetting`. A puzzle player reads the board without touching the screen, so the OS idle timer dims it mid-level.
5. **Logs: a policy enum keyed on `Debug.isDebugBuild`.** `LeaveDefault | DisableInReleaseBuilds | DisableInAllPlayerBuilds`. `DisableInReleaseBuilds` skips when `Debug.isDebugBuild` is true — which covers both the Editor and development builds — so **test builds keep their logs**. `DisableInAllPlayerBuilds` preserves the old (over-reaching) behaviour for anyone who wants it.
6. **The hook runs in the Editor too and applies its values.** `Application.targetFrameRate` is assertable in a PlayMode test that way (game-shikaku already has one), which keeps the feature testable. Documented caveat: the Game view has its own vsync, so the Editor's actual frame pacing is not governed by this — measure on device.
7. **No asset ⇒ nothing is touched.** Missing asset is the documented default and means "leave every platform default alone", satisfying the requirement not to force a rate on every consumer. Existing consumers who never create the asset see no behaviour change.

## API

Namespace `TK.Core.App` (existing `TK.Core.App` asmdef; no asmdef changes).

```csharp
public enum FrameRateMode { PlatformDefault, Fixed, MatchRefreshRate, HalfRefreshRate }
public enum SleepTimeoutPolicy { LeaveDefault, NeverSleep, SystemSetting }
public enum LogPolicy { LeaveDefault, DisableInReleaseBuilds, DisableInAllPlayerBuilds }

[Serializable]
public struct FrameRateProfile
{
    public FrameRateMode mode;
    [Min(1)] public int fixedFps;   // used only when mode == Fixed
}

[CreateAssetMenu(fileName = "TKStartupSettings", menuName = "TK/Startup Settings")]
public sealed class StartupSettings : ScriptableObject
{
    // serialized: mobile, standalone, sleepTimeout, logs
    public void Apply();                 // public so a PlayMode test can drive it directly
    public int ResolveTargetFrameRate(double refreshRateHz, bool isMobile);  // pure, unit-testable
}
```

- `StartupSettings.RESOURCES_NAME = "TKStartupSettings"` — the load key, exposed as a constant rather than a magic string.
- The `BeforeSplashScreen` hook lives in a small internal static class; it loads the asset and calls `Apply()`.

## Applier semantics

| Setting | Behaviour |
| --- | --- |
| `PlatformDefault` | `Application.targetFrameRate` is not written at all |
| `Fixed` | `= fixedFps` |
| `MatchRefreshRate` | `= round(Screen.currentResolution.refreshRateRatio.value)` |
| `HalfRefreshRate` | `= max(1, round(refresh / 2))` |
| `SleepTimeoutPolicy.LeaveDefault` | `Screen.sleepTimeout` not written |
| `NeverSleep` / `SystemSetting` | written to the matching `SleepTimeout` constant |
| `LogPolicy.LeaveDefault` | `Debug.unityLogger` untouched |
| `DisableInReleaseBuilds` | `if (!Debug.isDebugBuild) Debug.unityLogger.logEnabled = false` |
| `DisableInAllPlayerBuilds` | `logEnabled = false` in any player build (`#if !UNITY_EDITOR`) |

`ResolveTargetFrameRate` is a pure function of (refresh rate, isMobile) so the mode matrix — including the 90 Hz case — is covered by EditMode tests without a device. Unity 6 note: `Screen.currentResolution.refreshRate` (int) is deprecated; `refreshRateRatio` is the supported accessor — to be confirmed against the installed editor during implementation.

## Migration (game-shikaku, currently on core 0.4.0)

1. Delete the local `DisplayBootstrap` class. Keep the ADR as the record of the 60 FPS decision; repoint it at the package.
2. Create `Assets/Resources/TKStartupSettings.asset` (**Assets → Create → TK → Startup Settings**): `mobile = Fixed 60`, `standalone = PlatformDefault`, `sleepTimeout = NeverSleep`, `logs = DisableInReleaseBuilds`.
3. `AppBootstrapper.disableLogsInReleaseBuilds` is deprecated — move the intent to the asset. Behaviour change to be aware of: development builds now **keep** their logs (previously suppressed despite the field name).
4. The existing PlayMode test asserting `Application.targetFrameRate == 60` keeps passing, because the hook applies in the Editor.

## Explicitly out of scope

- `Application.runInBackground` — same family and the only plausible future addition, but on mobile the OS suspends the app anyway. Not added speculatively.
- `QualitySettings` level selection — a per-device-tier product decision with its own logic and UI; and `vSyncCount` is ignored on Android, so driving frame rate through quality levels is a trap.
- `Screen.orientation` — a Player Settings decision that games often change per scene; a startup override is a footgun.

## Verification

- **Harness (EditMode):** the pure `ResolveTargetFrameRate` matrix (all four modes × 60/90/120 Hz × mobile/standalone), the log-policy branch keyed on `isDebugBuild`, sleep-timeout mapping, and "no asset ⇒ nothing written".
- **Consumer (game-shikaku):** the existing PlayMode assertion on `targetFrameRate`, then an on-device build confirming the measured frame time stays at ~16.5 ms after removing the local `DisplayBootstrap`.
