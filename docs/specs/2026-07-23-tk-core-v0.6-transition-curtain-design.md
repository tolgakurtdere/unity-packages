# com.tk.core 0.6.0 — Design: TransitionCurtain (scene/state transition mask)

Approved by Tolgahan 2026-07-23. Driver: game-shikaku's level start/restart and menu return run
`SceneLoader.UnloadGameAsync()` / `LoadGameAsync(...)` with no visual mask — the screen hard-cuts in
one frame and half-built frames can leak. The *logical* gating already exists (`NavigationGate`
drops re-entrant flow verbs; tab taps drop on `IsTransitioning`); what's missing is the visual
cover and its raycast/back-input shadow. This was ROADMAP candidate #1 (`com.tk.transitions`);
the brainstorm resolved it into a `TK.Core.UI` addition instead of a standalone package.

## Locked decisions

1. **Separate component, not a TaskOverlay generalization.** TaskOverlay and the curtain have
   *opposite timing contracts*: TaskOverlay shows late (1 s grace so short waits never flash) and
   hides instantly; the curtain must be **fully opaque before the work starts** and animates open
   after. One class serving both would be a flag tangle. TaskOverlay is untouched; the two compose
   (curtain below, TaskOverlay's delayed spinner above).
2. **Lives in `TK.Core.UI`, ships as core 0.6.0 (additive minor).** Everything it uses is already
   there (UIManager, UICatalog, RefCountLock, back-input suppression). A standalone
   `com.tk.transitions` would be an 11th package whose whole content is sequencing three core
   primitives — rejected.
3. **Catalog prefab + code-generated fallback.** If the catalog has a `"TransitionCurtain"` entry,
   that prefab is used (game-supplied visual: logo, pattern, spinner — anything). If not, the
   package silently builds a plain black fade curtain in code. Unlike TaskOverlay's missing-key
   path, **no warning is logged** — the fallback is a legitimate default, not a broken setup.
4. **`RunAsync`-style wrapper is the main door; Show/Hide stay public for advanced flows.**
   The wrapper guarantees the curtain opens again in `finally`, so a throwing transition can never
   soft-lock the screen black (real risk: `NavigationGate.RunAsync` catches and logs exceptions,
   so nothing downstream would ever notice a stuck curtain).
5. **Name: `TransitionCurtain`.** Weighed against `ScreenFader` (most common term, but bakes the
   fade visual into the name — wrong for a logo/wipe view) and `TransitionOverlay` (collides with
   TaskOverlay's identity; "transition" is already carried by `IUITransition`,
   `NavigationGate.IsTransitioning`, `RunTransitionAsync`). "Curtain" is the minority-but-real
   Unity idiom (cf. the widespread `LoadingCurtain` pattern), and it names the role, not the visual.

## Components

### `TransitionCurtainView` (abstract MonoBehaviour, `TK.Core.UI`)

The visual contract:

```csharp
public abstract Awaitable ShowAsync();   // animate to FULL cover; on return the screen is 100% hidden
public abstract Awaitable HideAsync();   // animate open
```

Customization is two-tier — subclassing is only needed for new *animation behavior*:

- **Custom visual, fade animation (no code):** build a prefab with the stock `FadeCurtainView` at
  the root and any hierarchy under it (logo, pattern, background art) — the CanvasGroup fade
  covers the whole hierarchy. Register it under the `"TransitionCurtain"` catalog key. This is
  also how a project changes the defaults (duration/color): **the catalog prefab IS the config
  surface** — no separate ScriptableObject settings asset, same pattern as `"TaskOverlay"`. A
  Resources-SO on top would duplicate what the prefab's inspector already holds and create two
  fighting config sources.
- **Custom animation (script):** subclass `TransitionCurtainView`, implement
  `ShowAsync`/`HideAsync` (Animator, DOTween/PrimeTween, wipe, slide — anything), put it on a
  prefab, register it in the catalog. The one contract: when `ShowAsync` returns, the screen is
  fully hidden.

### `FadeCurtainView : TransitionCurtainView`

The dependency-free default: CanvasGroup alpha fade over a serialized duration (default 0.25 s)
and a serialized color (default black) on a full-screen `Image` (`raycastTarget = true`).
Differences from `DefaultPopupTransition`'s loop, both deliberate:

- **Unscaled time** (`Time.unscaledDeltaTime`) — transitions must animate even if the game pauses
  gameplay with `timeScale = 0` around a scene swap. (`DefaultPopupTransition` uses scaled time;
  that stays as-is — popups pausing with the game is defensible, the curtain freezing is not.)
- Keeps the zero-duration fully-synchronous jump path (never touches `Awaitable.NextFrameAsync`),
  which also makes the class safe to drive in EditMode tests.

### UIManager API (same placement pattern as the Task Overlay region)

```csharp
public const string TransitionCurtainKey = "TransitionCurtain";

/// Main door: closes the curtain, runs the work, reopens in finally.
/// Exceptions from work propagate (caller's NavigationGate logs them) — but the curtain
/// ALWAYS reopens, so a throwing transition cannot leave the screen black.
/// minCoverSeconds: optional minimum covered time (unscaled) — if the work finishes earlier,
/// the curtain stays closed until the timer elapses (branding/breathing room). Default 0 = open
/// as soon as the work completes. An exception skips the remaining hold and reopens immediately.
public async Awaitable RunUnderCurtainAsync(Func<Awaitable> work, float minCoverSeconds = 0f);

/// Advanced/manual control. Ref-counted: every Show must pair with one Hide;
/// the curtain visually opens only when the last holder releases.
public async Awaitable ShowCurtainAsync();
public async Awaitable HideCurtainAsync();
```

## Resolution & placement

- Lazy, on first use: catalog `TryGet(TransitionCurtainKey)` → Addressables instance via the
  existing `GetOrLoadAsync` path; else code-generate a `FadeCurtainView` (GameObject with
  full-screen black `Image` + CanvasGroup).
- Parented under `taskOverlayContainer` as **first sibling** → TaskOverlay always renders above
  the curtain. If work behind the curtain runs long (e.g. a CDN fetch during level start), the
  TaskOverlay's 1 s-delayed spinner appears *on top of* the curtain — the two compose without
  knowing about each other.
- The curtain instance lives with the persistent MainScene's UIManager, so it survives the very
  game-scene loads/unloads it masks.

## Behavior contract

- **Timing:** `RunUnderCurtainAsync` awaits `ShowAsync` *to completion* before starting the work —
  not a single swap frame can leak. After the work (and any remaining `minCoverSeconds`, unscaled),
  `HideAsync`. Even instant work with no min-cover still shows the full show+hide animation
  (~0.5 s smooth dip to black), so there is no flash in any configuration. Show/hide *durations*
  are view configuration (serialized on the prefab), never per-call parameters — a duration
  argument would have to thread through the abstract view seam and would fight the view's own
  animation design.
- **Reentrancy:** ref-counted via the existing `TK.Core.Utilities.RefCountLock`. Nested
  `RunUnderCurtainAsync` (or unbalanced-in-time Show/Show/Hide/Hide) shows once, hides when the
  last holder releases. Flow verbs are already serialized by `NavigationGate`, so nesting is rare
  but cheap to support.
- **Input:** the visual blocks raycasts by nature; keyboard/gamepad/Android back is covered by the
  existing `UIManager.PushBackInputSuppression()`/`PopBackInputSuppression()` around the covered
  period (paired exactly once per show/hide cycle, not per ref-count holder).
- **Errors:** `RunUnderCurtainAsync` reopens the curtain in `finally` and lets the exception
  propagate. If `ShowAsync` itself throws, the work does not run and the curtain is force-hidden.
- **Not a UIBase:** the curtain does not derive `UIBase`/`PopupBase` — it must not enter the
  back-button navigation stack, must not be closeable, and needs none of the layout/popup
  lifecycle. It is a plain MonoBehaviour view driven by UIManager.

## Untouched

`TaskOverlay`, `SceneLoader`, `NavigationGate`, all of `TK.Core.App` — zero changes. The App→UI
boundary stands: wiring the curtain into flow verbs is game-side composition.

## Consumer integration (game-shikaku, reference)

```csharp
// ShikakuAppFlow — already inside the transition lock; the curtain is its visual layer.
protected override Awaitable StartLevelAsync(int levelIndex) =>
    UIManager.Instance.RunUnderCurtainAsync(async () =>
    {
        _bindings.CancelTabNavigation();
        tabBar.SetVisible(false);
        GameplayLaunchContext.Set(/* unchanged */);
        await SceneLoader.UnloadGameAsync();
        await SceneLoader.LoadGameAsync(SceneLoader.GAME_SCENE);
        await _bindings.ShowGameplayHudAsync(levelIndex);
    });
// ShowMenuAsync wraps the same way.
```

## Verification

- **Harness (EditMode):** service logic against a fake `TransitionCurtainView` — ref-count
  (nested run → one Show/one Hide), exception in work still hides, exception in Show force-hides,
  exception skips the remaining min-cover hold, back-input suppression balanced, zero-duration
  `FadeCurtainView` reaches end state synchronously. (The timed min-cover wait itself is a
  player-loop `Awaitable` — its elapsing is PlayMode territory, like the fades.)
- **PlayMode (game-shikaku, acceptance):** Retry / Next Level / Return-to-menu each show a smooth
  fade instead of a hard cut; no input lands mid-transition; TaskOverlay still appears above the
  curtain when work is artificially delayed. The animated path is not harness-verifiable — the
  consumer round is the acceptance gate, per suite convention.
