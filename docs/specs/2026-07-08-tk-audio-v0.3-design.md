# com.tk.audio 0.3.0 — Design (music & settings polish)

Approved-in-principle by Tolgahan 2026-07-08 (v0.3 scope + behaviors pre-locked across the scoping conversation). This version is **behavior-changing** for the music channel's relationship to mute — not purely additive; see "Behavior change" below. SFX behavior is unchanged.

## Locked decisions (from the scoping conversation)

1. **Two mute paths, two behaviors:**
   - **Ads mute** (`PushMute`/`PopMute`) → **Pause** the music (freeze position; resume exactly on `PopMute`). Seamless: the player heard nothing during the ad, and music continues where it was.
   - **Settings mute** (`MusicEnabled = false`) → **Stop** the music. Turning it back on **restarts the remembered "desired music" from the top**. Booting with `MusicEnabled = false` → music never starts (the request is remembered, not played).
2. `PauseMusic()` / `ResumeMusic()` — explicit game-driven music pause (app-pause / phone-call), independent of ads mute; composes with it (music is paused while EITHER holds).
3. `FadeChannelVolume(channel, target, seconds)` — smooth a channel's volume (cutscene ducking / pause). Must NOT overwrite the durable saved volume — it drives a separate transient multiplier.
4. Settings `Changed` event — fired when any durable setting changes, so bound UI (a volume slider) reflects code-side changes.
5. `PreloadAsync(musicKey)` — warm up an addressable music clip before use to avoid the first-play hitch.

## Volume model (the core refactor)

Today `EffectiveMusicVolume = MusicEnabled && !IsMuted ? MusicVolume : 0` — both gates collapse to volume 0 (music plays silently). v0.3 splits music from that model:

- **Music channel** — a playing source's volume = `slotFadeWeight × entryScale × MusicVolume × _musicFade`. **No mute/enabled gate in the volume** because:
  - `MusicEnabled = false` → sources are **stopped** (not playing), so volume is moot.
  - ads-mute / manual pause → sources are **paused**, so volume is moot.
  - `_musicFade` (0..1, default 1, NOT persisted) is the transient `FadeChannelVolume` multiplier.
  - Exposed to the player as `AudioService.MusicPlayVolume` (`= MusicVolume × _musicFade`).
- **SFX channel** — unchanged mute semantics: `EffectiveSfxVolume = (SfxEnabled && !IsMuted ? SfxVolume : 0) × _sfxFade`. One-shots have nothing to pause; mute stays volume-gate + don't-spawn.
- **Music pause state** — `MusicPaused = IsMuted || _musicManuallyPaused`. When it changes, `MusicPlayer.SetPaused` calls `AudioSource.Pause()`/`UnPause()` on both slots AND the crossfade/track-monitor async loops freeze (don't advance elapsed while paused), so a mid-crossfade pause resumes seamlessly.

**Behavior change (one existing test updates):** `AudioServiceTests.EffectiveVolumes_CombineSettingsAndMuteStack` asserts music volume is gated to 0 by enabled/mute — that half is rewritten to the new pause/stop model. This is the only pre-existing test that changes; it's a deliberate, documented behavior change (same class as the tab bar hooks). Every other 0.2.0 test passes unmodified.

## Requested-music intent

`AudioService` remembers the last requested music (separate from what's playing): `None | Track(key|clip, loop) | Playlist(key)`.
- `PlayMusic`/`PlayPlaylist` set the request; `StopMusic` clears it.
- If `MusicEnabled`, the request plays immediately (unchanged UX); if not, it's only remembered.
- `MusicEnabled` false→true replays the request from the top; true→false stops the player but keeps the request.
- `MusicPlayer` stays mechanism-only (plays what it's told, pauses, crossfades); `AudioService` owns the policy (enabled/mute/pause/request).

## New API (all additive except the volume-model refactor)

- `void PauseMusic()` / `void ResumeMusic()` — manual music pause flag (composed into `MusicPaused`).
- `void FadeChannelVolume(AudioChannel channel, float target, float seconds)` — fades `_musicFade`/`_sfxFade` toward `target` (0..1) over unscaled `seconds` (0 = instant); generation-guarded per channel.
- `event Action Changed` — raised after any durable setting (`MusicEnabled`/`SfxEnabled`/`MusicVolume`/`SfxVolume`) changes.
- `Awaitable PreloadAsync(string musicKey)` — if the entry is addressable, loads + caches its handle so a later `PlayMusic` reuses the resident clip (the preload cache owns the handle; a slot playing a preloaded clip does NOT release it). Idempotent; released on `Dispose`.
- Static `Audio` façade mirrors: `PauseMusic`/`ResumeMusic`/`FadeChannelVolume`/`Changed`/`PreloadAsync`.

## Out of scope (later / a real consumer)
Level-scoped catalogs, addressable SFX + warm window, ducking presets, AudioMixer, named categories.

## Testing
EditMode: transient-fade math (`FadeChannelVolume(0)` instant → multiplier 0; the setting is untouched); `Changed` fires once per distinct setting change, not on no-op writes; ads-mute pauses music sources (`isPlaying` false while muted, true after `PopMute`, position preserved via the frozen loops — assert `time` doesn't advance across a paused frame span in a light play-mode-ish check or via the paused-flag state); `PauseMusic`/`ResumeMusic` compose with mute (paused if either holds); settings-mute stops + `MusicEnabled=true` replays the remembered request from the top (`ActiveMusicKey` restored); boot-muted never starts (`ActiveMusicKey` null, request remembered); `PreloadAsync` state (preloaded key reused, handle released on Dispose). The updated `EffectiveVolumes` test reflects the new model. Crossfade/pause frame-driven paths are play-mode verified in game-shikaku before tagging.
