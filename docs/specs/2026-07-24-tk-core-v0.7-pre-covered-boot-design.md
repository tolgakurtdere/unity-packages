# com.tk.core 0.7.0 — Design: pre-covered boot (instant curtain cover)

Approved by Tolgahan 2026-07-24. Driver: game-shikaku's 0.6.0 acceptance observed the "boot
breath" — with the boot menu show wrapped in `RunUnderCurtainAsync`, the player watches
open → cover → open at app start. Recorded in ROADMAP as a consumer-confirmed, brainstorm-gated
feature; this is that brainstorm's outcome.

## The boot-sequencing fact the design rests on

`AppBootstrapper.Start` (SplashScene) waits the splash delay, then `LoadMainSceneAsync` loads
MainScene **additively** — at which point the game flow's `Awake`/`Start` (and therefore
`OnBootAsync`) begin running **while the splash scene is still loaded**; the splash unload
starts after and takes at least a frame. Two consequences:

1. The splash window is the cover window: a curtain covered inside it is revealed
   already-opaque when the splash goes — the player never sees a closing animation.
2. The breath is not only cosmetic: missing that window can also leak half-built MainScene
   frames between splash removal and menu setup. Pre-covered boot kills both.

## Locked decisions

1. **Full guarantee, not just hidden close.** The contract is "splash → one animated curtain
   reveal": cover instantly inside the splash window, build the menu behind it, animate open
   once. Not the weaker "first show skips its animation" — same mechanism, stronger promise.
2. **Catalog prefab loads async inside the splash window.** When a `"TransitionCurtain"`
   catalog prefab exists, instant boot cover awaits its (typically 1–2 frame, local
   Addressables) load, then snaps covered. Visual consistency wins: the boot cover is the same
   curtain every other transition uses. Residual risk (a load slower than the splash unload)
   is documented, not engineered away — the rejected alternatives were "always sync black at
   boot" (visual mismatch with a logo curtain) and a two-layer sync-black-then-swap hybrid
   (two-view lifecycle complexity, not v1-worthy). **The no-catalog fallback path completes
   synchronously in the same frame** — no Addressables, no frame yield — so for games without
   a curtain prefab (shikaku today) the guarantee is absolute.
3. **Hold-based API: `CoverCurtainInstantlyAsync()` paired with `HideCurtainAsync()`.** One
   new door that rides the existing ref-count: it takes a hold like `ShowCurtainAsync`, just
   without the animation. The game covers at the top of `OnBootAsync` and releases in
   `finally`; every transition in between runs its normal `RunUnderCurtainAsync` unchanged
   (already covered → its Show returns instantly, its Hide drops the count to 1 and stays
   covered). Rejected: an `instantCover` flag on `RunUnderCurtainAsync` (shikaku's
   `ShowMenuAsync` serves both boot and menu-return, so the game would have to thread an
   "am I booting" bit, and cover timing would be locked to the run call), and
   "first-ever Show is auto-instant" (magic; wrong default for games that don't curtain boot).

## Components

### Seam addition: `ShowInstantly()` (the 0.7.0 version driver)

`ITransitionCurtainView` gains a third member; `TransitionCurtainView` declares it abstract:

```csharp
void ShowInstantly();   // snap to FULL cover synchronously — no animation, no frame yield
```

`FadeCurtainView` implements it in two lines (`blocksRaycasts = true; alpha = 1f`). Abstract,
not virtual-with-default: any "default" would either animate (lying about the contract) or
guess at the subclass's visual state. This breaks raw-interface implementers and
`TransitionCurtainView` subclasses — the seam is one day old and its only known implementers
are this package and its test fakes; same seam-break-in-a-minor precedent as
`com.tk.notification` 0.2.0 / `com.tk.haptics` 0.2.0. For consumers of the shipped views the
release is purely additive.

### `TransitionCurtainController.CoverInstantlyAsync()`

Rides the existing Show machinery — same holds, same waiters, same suppression callbacks, same
failure path. Implementation shape: `ShowAsync` and `CoverInstantlyAsync` share one core that
sets a one-shot pending-instant flag; the settle loop's cover branch consumes it and calls
`view.ShowInstantly()` instead of `await view.ShowAsync()`. Everything after (state flip,
waiter flush) is identical, and the open side is always animated — instant applies to covering
only.

Edge contracts (documented, tested where cheap):

- Instant request while an **animated cover is already in flight** joins that cover (no
  mid-animation fast-forward; cannot occur at boot, not worth the machinery).
- Instant request while **already covered** = one more hold, returns immediately (same as
  `ShowAsync`).
- If several Show demands coalesce into one cover pass and ANY of them asked for instant, that
  cover is instant — the flag is consumed per cover, not per caller.
- Failure path unchanged: a throwing resolve/`ShowInstantly` fails the queued waiters,
  force-opens, drops all holds.

### UIManager

```csharp
/// Covers the curtain WITHOUT animation and takes one hold — call at boot while the splash
/// still covers the screen, release with HideCurtainAsync. On the no-catalog fallback path
/// this completes synchronously in the same frame.
public Awaitable CoverCurtainInstantlyAsync() => Curtain.CoverInstantlyAsync();
```

## Consumer integration (game-shikaku, reference)

```csharp
protected override async Awaitable OnBootAsync()
{
    await UIManager.Instance.CoverCurtainInstantlyAsync();   // splash still up; same-frame cover
    try { await base.OnBootAsync(); }                        // menu builds behind the curtain
    finally { await UIManager.Instance.HideCurtainAsync(); } // the single animated reveal
}
```

`ShowMenuAsync`'s existing `RunUnderCurtainAsync` wrapper stays byte-identical.

## Untouched

`TK.Core.App` (including `AppBootstrapper`/`AppRootBase`), `SceneLoader`, `TaskOverlay` — zero
changes. The App→UI boundary stands: boot cover is game-side composition in `OnBootAsync`.

## Verification

- **Harness (EditMode):** fake view gains a `ShowInstantlyCalls` counter — instant cover calls
  `ShowInstantly` and never `ShowAsync`; instant → nested `RunAsync` → final Hide chain shows
  one push/one pop and exactly one animated `HideAsync`; instant-while-covered adds a hold with
  no extra view call; coalesced instant+animated demand covers instantly; failure path parity.
  `FadeCurtainView.ShowInstantly` reaches alpha 1 / blocking synchronously. Fallback-path
  synchronous completion: `CoverCurtainInstantlyAsync` with a synchronously-resolving view
  completes without a frame yield (assertable in EditMode by observing completion before any
  pump).
- **PlayMode (game-shikaku, acceptance):** boot shows no closing animation — splash gives way
  to an already-covered curtain and one animated reveal; no half-built menu frame visible
  (frame-sampling probe: MainScene first visible frame has curtain alpha = 1); menu-return and
  level transitions unchanged (regression).

## Versioning
