# com.tk.core v0.5.0 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:executing-plans (inline). Binding contracts live in `docs/specs/2026-07-22-tk-core-v0.5-startup-settings-design.md` — READ IT FIRST. The whole point of this feature is *when* it runs: at `BeforeSplashScreen`, before any scene exists. Any design that needs a scene object has missed the requirement.

**Goal:** Let a project own its startup platform policy — target frame rate, sleep timeout, and log suppression — from one asset applied before the splash screen, instead of a splash-scene `MonoBehaviour` that runs too late.

**Architecture:** A `Resources`-loaded `StartupSettings` ScriptableObject. A `[RuntimeInitializeOnLoadMethod(BeforeSplashScreen)]` hook loads it and calls `Apply()`. All decision logic lives in **pure static resolvers** so the mode matrix is unit-testable without touching global Unity state or a device. No asset ⇒ nothing is written.

**Tech Stack:** Unity 6000.3.6f1 harness, NUnit EditMode. Existing `TK.Core.App` asmdef — no asmdef or dependency changes.

### Task 1: Config types + pure resolvers + the test matrix

**Files:** create `Runtime/App/FrameRateProfile.cs` (`FrameRateMode` enum + `FrameRateProfile` struct), `Runtime/App/SleepTimeoutPolicy.cs`, `Runtime/App/LogPolicy.cs`, `Runtime/App/StartupSettings.cs`; create `Tests/Editor/StartupSettingsTests.cs`.

- Enums exactly as the spec: `FrameRateMode { PlatformDefault, Fixed, MatchRefreshRate, HalfRefreshRate }`, `SleepTimeoutPolicy { LeaveDefault, NeverSleep, SystemSetting }`, `LogPolicy { LeaveDefault, DisableInReleaseBuilds, DisableInAllPlayerBuilds }`. `[Serializable] struct FrameRateProfile { public FrameRateMode mode; [Min(1)] public int fixedFps; }`.
- `StartupSettings : ScriptableObject`, `[CreateAssetMenu(fileName = RESOURCES_NAME, menuName = "TK/Startup Settings")]`, `public const string RESOURCES_NAME = "TKStartupSettings"`. Serialized: `mobile` (default `Fixed`/60), `standalone` (default `PlatformDefault`), `sleepTimeout` (default `LeaveDefault`), `logs` (default `DisableInReleaseBuilds`).
- Three **pure statics** — this task ships only these plus the type shapes:
  - `public static int? ResolveTargetFrameRate(FrameRateProfile profile, double refreshRateHz)` — `PlatformDefault` → `null` ("do not write"); `Fixed` → `max(1, fixedFps)`; `MatchRefreshRate` → `max(1, round(refreshRateHz))`; `HalfRefreshRate` → `max(1, round(refreshRateHz / 2))`. `null` is the sentinel, **not** `-1` — `-1` is a real Unity value and would be indistinguishable from "leave alone".
  - `public static SleepTimeout? ResolveSleepTimeout(SleepTimeoutPolicy policy)` — `LeaveDefault` → `null`, else the matching `SleepTimeout` constant.
  - `public static bool ShouldDisableLogs(LogPolicy policy, bool isDebugBuild, bool isPlayerBuild)` — `LeaveDefault` → false; `DisableInReleaseBuilds` → `isPlayerBuild && !isDebugBuild`; `DisableInAllPlayerBuilds` → `isPlayerBuild`.
- Tests (all pure, no global state touched):
  - Frame-rate matrix: `PlatformDefault` → null at every refresh; `Fixed 60` → 60; `MatchRefreshRate` → 120/90/60 at 120/90/60 Hz; `HalfRefreshRate` → 60/45/30 at 120/90/60 Hz. **The 90 Hz row is the reason these modes exist — assert it explicitly.** Degenerate refresh (0) still yields ≥ 1.
  - Log matrix: Editor (`isPlayerBuild:false`) never disables under any policy; development build (`isDebugBuild:true, isPlayerBuild:true`) keeps logs under `DisableInReleaseBuilds` but loses them under `DisableInAllPlayerBuilds`; release build (`isDebugBuild:false, isPlayerBuild:true`) loses them under both.
  - Sleep mapping tri-state.
  - A freshly `CreateInstance<StartupSettings>()` asset comes up with the documented defaults (guards the "new asset is sane" property).

### Task 2: The BeforeSplashScreen hook + graceful `AppBootstrapper` deprecation

**Files:** modify `Runtime/App/StartupSettings.cs` (add `Active`, `Apply()`, the hook), modify `Runtime/App/AppBootstrapper.cs:21-32`.

- `public static StartupSettings Active { get; private set; }` — lets other code (and `AppBootstrapper`) tell whether an asset took over.
- `public void Apply()`: set `Active = this`; pick `Application.isMobilePlatform ? mobile : standalone`; write `Application.targetFrameRate` only when the resolver returns a value; write `Screen.sleepTimeout` only when the resolver returns a value; disable `Debug.unityLogger.logEnabled` when `ShouldDisableLogs(logs, Debug.isDebugBuild, IsPlayerBuild)` — where `IsPlayerBuild` is `false` under `UNITY_EDITOR` and `true` otherwise.
- The hook:
  ```csharp
  [RuntimeInitializeOnLoadMethod(RuntimeInitializeLoadType.BeforeSplashScreen)]
  private static void LoadAndApply()
  {
      Active = null;                                        // domain-reload-off safety, same pattern as UIManager/SceneLoader
      var settings = Resources.Load<StartupSettings>(RESOURCES_NAME);
      if (settings == null) return;                         // documented: no asset ⇒ nothing is touched
      settings.Apply();
  }
  ```
- `AppBootstrapper`: keep the `disableLogsInReleaseBuilds` field but make it a **fallback**, so consumers with no asset see no behaviour change:
  ```csharp
  #if !UNITY_EDITOR
      if (StartupSettings.Active == null && disableLogsInReleaseBuilds) Debug.unityLogger.logEnabled = false;
  #endif
  ```
  Update its `[Tooltip]` to say it is deprecated and superseded by `TKStartupSettings`' `LogPolicy` (which, unlike this field, leaves development builds alone). Do **not** delete the field — removing it would silently re-enable logs in existing consumers' release builds.
- No test asserts `Apply()`'s global writes in EditMode (it would leak state across the suite); the decision logic is already covered by Task 1's pure resolvers, and the applied value is asserted by the consumer's PlayMode test.
- [ ] Implement + gate + commit `feat(core): apply startup settings at BeforeSplashScreen; deprecate the AppBootstrapper log flag`.
